# Участие в разработке data-vac-frontend

## Именование веток

Все ветки должны начинаться с одного из префиксов:

| Префикс | Когда использовать |
|---|---|
| `feat/` | Новая функциональность |
| `fix/` | Исправление багов |
| `chore/` | Тех. задачи / обслуживание |
| `docs/` | Документация |
| `refactor/` | Рефакторинг без изменения логики |

**Примеры:** `feat/infections-page`, `fix/search-bar`, `refactor/routing`

## Начало работы

Перед созданием ветки всегда синхронизируемся с `main`:

```bash
git checkout main
git pull origin main
git checkout -b feat/your-task-name
```

## Сообщения коммитов

Репозиторий использует [Conventional Commits](https://www.conventionalcommits.org/).

```
feat(infections-page): добавить фильтрацию по категориям
fix(search-bar): исправить потерю фокуса при вводе
chore: обновить зависимости
```

Типы: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `style`, `perf`

## Перед открытием PR

Синхронизируемся с последним `main` через rebase (не merge):

```bash
git fetch origin
git rebase origin/main
```

Если возникли конфликты — разбираемся в каждом файле отдельно. Если код непонятен — сначала уточняем у автора. После разрешения:

```bash
git add .
git rebase --continue
```

## Пуш ветки

```bash
git push origin feat/your-task-name

# Если история ветки была переписана через rebase:
git push origin feat/your-task-name --force-with-lease
```

## Правила Pull Request

- PR — **единственный** способ попасть в `main`, прямой пуш заблокирован
- Заголовок PR должен соответствовать формату Conventional Commits
- Базовая ветка всегда `main`

## Структура Pull Request

**Заголовок** — в формате Conventional Commits:
```
feat(infections-page): добавить страницу списка инфекций
```

**Описание:**

```markdown
## Что сделано

- Реализована страница `InfectionsPage` с двухколоночной сеткой
- Подключён фильтр по категориям через `TagFilter`
- Счётчик результатов обновляется при фильтрации
- Роутинг подключён

## Как проверить

1. Запустить приложение: `npm run dev`
2. Перейти на страницу инфекций
3. Проверить фильтрацию по категориям
4. Проверить счётчик результатов

## Связанные задачи

Closes #24
Closes #23
```

> `Closes #24` автоматически закроет issue и создаст двустороннюю ссылку при мерже в `main`.
> Если одна ветка закрывает несколько задач — перечислить каждую на отдельной строке.

**Чеклист перед отправкой на ревью:**
- [ ] Ветка синхронизирована с `main` через `rebase`
- [ ] Приложение запускается без ошибок
- [ ] Проверен в браузере на десктопе и мобильном

---

## Стратегия слияния

- **Squash merge** — предпочтительный вариант (история `main` остаётся чистой)
- **Rebase merge** — если важна линейная история отдельных коммитов
